# Classifyre — Product Improvement Recommendations (Enron Second Attempt)

Build/env: Classifyre desktop **v0.4.60**, namespace `ns_eron_email_no_2`, API `http://localhost:9933`. Second Enron-corpus smoke test; see `docs/first-use/enron-second-attempt/bugs.md` for full bug detail and `docs/first-use/enron/` for the first attempt's docs. Every recommendation below is tied to a specific bug/observation from this run; nothing here is invented.

---

| R-1 | LLM detector errors (503/429 from provider) are caught and swallowed as an empty result; `detector_outcomes` records `OK` on 98/100 assets and the run finishes `COMPLETED` with zero LLM findings — indistinguishable from a genuine "found nothing." | An investigator relying on a custom LLM detector at scale gets silent total failure with a green run status. Trust-destroying: the product cannot tell the difference between "no findings" and "detector never ran." | (a) In `apps/cli/src/detectors/custom/runners/_llm.py` `_complete_and_parse`, stop catching-and-swallowing provider exceptions; propagate them into `detector_outcomes` as `ERROR`/`WARNING`, never `OK`. (b) Throttle per-worker LLM concurrency and add backoff that respects the provider's own advertised rate limit (observed hitting 186 requests against a 48 limit). (c) Roll run status up: a run where a configured detector never successfully executed on any asset should not report plain `COMPLETED`. | Restores the basic contract that a completed run's "no findings" means something was actually checked. Directly fixes the most severe bug of this test. | **P0** | detectors / CLI | BUG A | Related to G-014 (same failure class: crashed detector reports success), not a literal regression |
| R-2 | Agent runs and rows for a different namespace (`ns_eron_mail`) are present and executing inside a freshly created namespace (`ns_eron_email_no_2`). | Namespace isolation — a basic multi-tenant/multi-project guarantee — does not hold. Provenance in a "clean" namespace is polluted by foreign runs; in a write-enabled configuration this could mean an agent applies decisions based on another namespace's events. | Scope the autopilot scheduler's event/source queries by namespace at the query layer (not just at the MCP read-tool layer, which does correctly 404). Add an integration test that creates two namespaces and asserts no agent_run in namespace A ever references a sourceId/runnerId from namespace B. | Closes a namespace-isolation hole before it becomes a write-path incident; restores trustworthy per-namespace provenance. | **P0** | autopilot | BUG D | New |
| R-3 | The single configured LLM detector executes twice per asset because its `get_supported_content_types` returns both text and vision types, so it lands in more than one bucket of `all_active = text_detectors + binary_detectors + link_detectors` with no dedup. | 2x LLM cost and 2x rate-limit pressure for no benefit, directly compounding R-1's rate-limit failures. | Deduplicate `all_active` by detector identity before dispatch in `apps/cli/src/pipeline/detector_pipeline.py` (~line 163), or make bucket membership mutually exclusive per detector instance rather than per content-type match. | Halves LLM spend for any detector matching multiple content-type buckets; reduces the rate-limit pressure that produces BUG A. | **P1** | detectors / CLI | BUG B | New |
| R-4 | MCP `create_detector_test_scenario` documents a nested `expected_outcome` shape that the server comparator does not read; the comparator instead expects an undocumented flat shape with underscore→space label normalization. A detector that fires correctly (0.99 confidence, quote+rationale populated) shows `FAIL`. | Anyone following the documented tool schema writes test scenarios that always fail, masking real detector regressions behind noise and eroding trust in the test suite. Debugging requires reverse-engineering the comparator source. | Make the comparator accept the documented nested shape (or fix the docs to match the actual flat shape) and document the underscore/space label normalization explicitly in the tool description. Also emit an expected-vs-actual diff on FAIL, and support re-running a single scenario instead of forcing an N-LLM-call full re-run every time. | Test scenarios become trustworthy and cheap to iterate on; removes a documented-but-wrong footgun. | **P1** | MCP / API | BUG C | Delete-scenario gap = G-010 (unchanged); schema/label mismatch is new |
| R-5 | Autopilot CASE agent runs intermittently fail with "OpenAI model not found (404)" while INQUIRY/DETECTOR_AUTHOR agents and the manual LLM detector succeed on the same provider concurrently. | Case-building automation silently drops runs; an investigator relying on autopilot to surface cases may miss ones that failed this way, with no indication surfaced outside the raw agent_runs table. | Audit CASE-agent-specific model/provider configuration for a hardcoded or stale model name distinct from the other agent kinds; add explicit alerting/visibility when an agent kind's failure rate diverges from its siblings. | Removes an intermittent automation failure mode; prevents cases from silently not being built. | **P1** | autopilot | BUG E | New |
| R-6 | MCP has no autopilot tools at all — every autopilot observation, run, and decision in this test was retrieved via raw REST `/autopilot/*` calls, not MCP. Separately, the CONFIG agent silently enabled a SECRETS detector on an operator-authored source and triggered its own rescan, with no consent surface presented to the operator. | An MCP-only operator (the product's intended integration surface) currently cannot see, audit, or control autopilot behavior at all — including autonomous config changes and rescans the agent decided to make on its own. This is both a visibility gap and a governance gap: autopilot can change detector configuration and trigger scans without any approval step. | Add MCP tools mirroring `/autopilot/*` (list/get runs, list decisions, list memories) so autopilot state is inspectable from the same surface as everything else. Separately, add a consent/approval gate (or at minimum a loud, queryable "pending confirmation" state) before CONFIG-agent-initiated detector changes take effect and before they trigger a rescan — do not let config mutation + rescan happen atomically and silently. | Closes the MCP capability gap (first raised as G-005) and adds a control point for autonomous configuration changes, which is now demonstrated to actually happen unprompted, not just theoretically possible. | **P1** | MCP / autopilot | Context notes (CONFIG agent auto-enabling SECRETS detector + rescan); MCP gap = G-005 | MCP gap = G-005 (unchanged); CONFIG-agent auto-mutation observation is new |
| R-7 | Every CRITICAL-severity finding in this run was a UK_NHS recognizer false-positive on US phone numbers / a bare timestamp, all at confidence 1.0; the evidenceAnalysis ranking layer correctly tags it "neutral" but still scores it importance 0.856 as "unique evidence." Same lesson as first attempt (there it was CREDIT_CARD on phone/OCR artifacts). | Investigators triaging by severity get led to false positives first, repeatedly, across two independent probes and two different recognizers. This is a systemic scoring problem, not a one-off recognizer bug. | Region-aware or corpus-aware recognizer selection (don't apply UK_NHS-format matching to a US corpus by default), and feed the evidenceAnalysis "neutral severity" signal back into the importance/ranking score, not just a separate label — a neutral-severity FP should not out-rank genuine evidence. | Breaks the recurring "severity ≠ evidence quality" failure pattern that has now shown up twice with two different PII recognizers. | **P1** | findings / detectors | Product-judgment: Severity ≠ evidence quality (replicated) | Same lesson replicated with a different recognizer (first attempt: CREDIT_CARD) |
| R-8 | Built-in PII ORGANIZATION extraction mixes real orgs (Sunoco, FERC, Bracewell & Patterson) with junk spans ("Definitive Agreement", "Grand Total", newline-glued multi-line spans). | Low precision forces manual review of every ORGANIZATION finding, undercutting the value of automated extraction for this entity type. | Add span-quality filtering (reject spans crossing newlines without a delimiter, filter a stoplist of generic legal/financial boilerplate phrases) before surfacing ORGANIZATION findings, or expose a confidence/quality score investigators can filter on. | Reduces manual triage load on the most commonly needed entity type in a corporate-email corpus. | **P2** | findings / detectors | Product-judgment: ORGANIZATION extraction noisy | New |
| R-9 | Semantic search results carry no snippet, chunk reference, or per-result score — only the ranking mode is echoed back. | An investigator cannot verify or judge relevance of a semantic search result from the product's own output; they must go elsewhere to confirm a hit is real. | Return the matched snippet/chunk text and a similarity score alongside each semantic search result. | Makes semantic search results self-verifying, closing a basic usability gap in the product's core search feature. | **P2** | semantics | Product-judgment: semantic search no snippet/score | New |
| R-10 | LOCAL_FOLDER RANDOM sampling is capped at 100 objects (documented as `rows_per_page`, "tabular only") and is seed-0 deterministic, so repeated "random" samples return the identical 100 files every time. | Users who re-run a "random sample" expecting a fresh look at the corpus silently get the same subset every time; the cap is also undocumented as applying to non-tabular LOCAL_FOLDER sources. | Either raise/parameterize the sample cap for LOCAL_FOLDER or document the 100-object ceiling explicitly in the tool description; make the seed vary per invocation unless a fixed seed is explicitly requested (for reproducibility use cases). | Removes a silent, surprising default that undermines what "random sample" means to a user. | **P2** | ingestion / docs | Product-judgment: RANDOM sampling cap+determinism | Yes — G-002 (cap, "tabular only") / G-003 (seed-0 determinism), unchanged |
| R-11 | `get_asset` reports status `"UNCHANGED"` for an asset on the very run that created it, while the run-level summary correctly reports `assetsCreated: N`. | Confusing/contradictory signals between asset-level and run-level status for the most common case (first sighting of a new file). | Have `get_asset` derive status from the same `change_type` computation the run summary uses, so a first-seen asset reports `CREATED` (or equivalent), not `UNCHANGED`. | Removes a small but confidence-eroding inconsistency between two views of the same fact. | **P2** | ingestion / API | Product-judgment: misleading asset status on first sighting | New |
| R-12 | The autopilot CONFIG agent overwrote an operator's just-saved detector configuration (LLM-only → PII+SECRETS) 13 seconds after the operator saved it, using a stale pre-edit base, and then executed on the operator's own manually-triggered run. No conflict detection, consent, or notification. | An operator cannot trust that a configuration they saved will be the one that runs — the agent can silently replace it in the gap before a run starts. This directly corrupted a controlled experiment in this test and, in real use, means detector selection is not under reliable operator control while autopilot is on. | Add optimistic-concurrency/version checks to source-config writes so an agent cannot silently clobber a newer operator write; require a confirmation or emit a queryable "pending config change" state before a CONFIG-agent mutation takes effect; never bundle config-mutation + auto-rescan atomically without a control point. | Restores operator authority over configuration; prevents autopilot from executing a config the operator did not choose. | **P0** | autopilot | BUG F | Related to first-attempt OBS-12 (autopilot mutated operator config), escalated here to a concurrent-edit race |
